CodeZen Explained as a Real Restaurant

Imagine this backend as a modern restaurant where customers place food orders. The server takes orders quickly, the kitchen processes them in sequence, and each dish is cooked in its own safe station. This page maps every file, dependency, and data flow to that real-life process in depth.

3) Request Flows As Real Restaurant Scenarios

Flow A: Quick Ticket + Track Later (/submit + /result)

  1. Customer places an order at counter: POST goes to main.py.
  2. Counter checks menu rules in schemas.py.
  3. Counter puts a kitchen ticket on conveyor via run_code_task.delay(...).
  4. Customer gets a ticket ID immediately (fast response).
  5. Kitchen worker in tasks.py picks ticket from Redis queue.
  6. Worker sends order to station in executor.py.
  7. Station runs code in temporary sandbox container and captures output.
  8. Result stored in Redis backend.
  9. Customer asks status with /result/{task_id} and gets queued or completed info.

Flow B: Place Order and Wait at Counter (/execute)

  1. Customer places order and asks to wait for result now.
  2. main.py still queues the task using Celery.
  3. Counter waits up to wait_timeout seconds.
  4. If ready in time, full result is returned in one response.
  5. If not ready, status stays queued and customer can track later with ticket ID.

5) Deep Focus: What Exactly Happens Inside executor.py

In the analogy, this is the secure cooking booth where one dish is prepared in isolation.

  1. Normalize language input (example: js becomes javascript).
  2. Select command and source filename from language config.
  3. Create a Docker client and isolated container with strict limits:
  4. Memory limit 256m, CPU cap, process limit, no network, no-new-privileges, non-root user.
  5. Package source code and stdin file into tar archive.
  6. Copy archive into container sandbox directory.
  7. Start container and wait for completion up to timeout.
  8. If timeout happens, kill container and mark timed_out true.
  9. Collect stdout and stderr logs from container.
  10. Truncate very large outputs to prevent huge payload response.
  11. Remove container forcefully and close Docker client in final cleanup block.
  12. Return structured result object: stdout, stderr, exit_code, timed_out, duration_ms.

6) Error and Edge Case Coverage

Situation Where Handled What User Sees
Unsupported language schemas.py validate_language Validation error with supported values list
Task creation failure main.py submit or execute routes HTTP 500 with failure detail
Long-running code executor.py timeout branch timed_out true, exit_code 124, timeout message in stderr
Docker runtime error executor.py APIError or DockerException catch stderr starts with Docker sandbox error
Result not ready yet main.py /result and /execute timeout handling Status stays queued or running
Huge output spam executor.py _truncate_output Output clipped with output truncated marker

7) Terminology Glossary in Plain Language

TaskOne unit of background work executed by Celery worker.
QueueWaiting line where tasks sit before workers pick them.
BrokerMessage transport service. Here it is Redis.
Result backendStorage where completed task states and outputs are kept.
WorkerBackground process that executes queued tasks.
SandboxIsolated container that runs untrusted user code safely.
stdinInput stream fed to the user program.
stdoutNormal printed output from program.
stderrError output from program or runtime.
Exit codeNumeric process status where 0 usually means success.
SerializationConverting payload to JSON for queue transport.
OrchestrationStarting and linking multiple services together.

8) Practical Debug Order (When Something Breaks)

  1. Check API route behavior in main.py first.
  2. Verify input schema assumptions in schemas.py.
  3. Confirm queue setup and Redis URL in celery_app.py.
  4. Check worker task mapping in tasks.py.
  5. Inspect Docker availability and sandbox image for executor.py.
  6. Validate service wiring and Docker socket mount in docker-compose.yml.
  7. Rebuild images if dependency mismatch appears in requirements.txt or Dockerfiles.
