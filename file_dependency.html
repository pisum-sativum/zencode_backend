2. The Journey of a Request (Step-by-Step Flow)

When a user sends code to the backend, here is exactly how the files hand off the work:

  1. Step 1 (main.py): The request arrives at the /submit endpoint.
  2. Step 2 (schemas.py): main.py uses ExecuteRequest to validate the input (ensuring the language is supported and code isn't empty).
  3. Step 3 (tasks.py & celery_app.py): main.py calls run_code_task.delay(). This packages the request and drops it into the Redis queue using settings from celery_app.py.
  4. Step 4 (tasks.py): The background Worker picks up the message from Redis and triggers the run_code_task function inside tasks.py.
  5. Step 5 (executor.py): The task passes the code to execute_code() in executor.py.
  6. Step 6 (Docker): executor.py talks to the Docker daemon to spin up a temporary sandbox container, runs the code, captures the output, destroys the container, and returns the result back up the chain.

3. In-Depth Breakdown of Each File

main.py The API Receptionist

Role: This is the entry point for the FastAPI web server. It listens for HTTP requests from the outside world.

Dependencies: It relies heavily on schemas.py to know what data to accept, and tasks.py / celery_app.py to hand off heavy work so the receptionist doesn't get blocked.

  • What's inside: Route definitions (@app.post("/submit"), @app.get("/result")).
  • Why it's important: It parses HTTP traffic into Python objects and manages the quick replies (like "Your task is queued in Celery!").

schemas.py The Rulebook

Role: Defines the shape, size, and rules of the data entering and leaving the system using Pydantic models.

Dependencies: None (it's pure logic and definitions). It is imported by main.py.

  • What's inside: Classes like ExecuteRequest and TaskResultResponse. It also contains the list of supported languages (Python, C, C++, Java, JS).
  • Why it's important: If a user tries to send an unsupported language like "Ruby", schemas.py throws an error before the heavy system even wakes up.

celery_app.py The Conveyor Belt Config

Role: Sets up the connection to Redis and configures how background jobs should be handled.

Dependencies: Needs a running Redis instance (URL provided via environment variable).

  • What's inside: The Celery(...) instance creation and configuration settings (like using JSON for serializing data).
  • Why it's important: Both main.py (the sender) and tasks.py (the receiver) need this exact same instance to know how to talk to each other through Redis.

tasks.py The Worker Instructions

Role: Acts as the bridge between the Celery queue system and the actual execution engine.

Dependencies: Imports the celery_app to register itself as a background task, and imports executor.py to do the actual work.

  • What's inside: The run_code_task function decorated with @celery_app.task.
  • Why it's important: It takes the raw dictionary payload pulled from Redis and passes the right fields (language, code, stdin) cleanly into the Python execution engine.

executor.py The Factory Machine

Role: The heart of the application. It takes the code, puts it in a secure virtual box (sandbox), runs it, and extracts the output.

Dependencies: Heavily relies on the installed docker Python package and needs access to the host's Docker socket.

  • What's inside: execute_code(). It has logic to create temporary tar files (to copy code into the container), applies strict memory and CPU limits, and listens to the container's stdout and stderr streams.
  • Why it's important: This ensures a malicious user cannot break the server (e.g., by writing a basic infinite loop or attempting to delete system files). It sandboxes the execution.

docker-compose.yml & Dockerfiles The Infrastructure

Role: These files are not Python code, but they run the Python code. They wire all the isolated parts together.

  • Dockerfile: Defines the environment for main.py and tasks.py. It installs FastAPI and Celery.
  • Dockerfile.sandbox: Creates the environment for exactly what executor.py runs. It contains the compilers (GCC, JDK, Node.js, Python).
  • docker-compose.yml: Boots up Redis, the API server, and the Celery Worker server simultaneously so they can instantly talk to each other.
